# Quantum ML Control Center Frontend

Next.js App Router frontend for job submission, queue monitoring, and results visualization.

## Local setup

1. Install dependencies:

	npm install

2. Copy environment template:

	copy .env.example .env.local

3. Run the frontend:

	npm run dev

4. Open:

	http://localhost:3000

## Implemented in this first slice

- API routes:
  - GET /api/health
  - GET /api/datasets
  - GET /api/jobs
  - POST /api/jobs
  - GET /api/jobs/:id
- Shared contract schemas using Zod
- Dashboard page for creating and polling jobs
- Postgres-backed job metadata when `POSTGRES_URL` is configured
- Local JSON-file fallback metadata when Postgres is not configured
- Queue mode via QStash when `QSTASH_URL` and `QSTASH_TOKEN` are configured
- Direct worker dispatch fallback when queue settings are absent
- Authenticated CSV uploads and csv_upload job execution path

## Queue mode setup

Set these in `.env.local` or Vercel project settings:

- `QSTASH_URL`
- `QSTASH_TOKEN`
- `APP_BASE_URL` (public URL of the frontend API)
- `WORKER_WEBHOOK_SECRET` (used for internal dispatch authorization)

When queue mode is enabled, jobs are published to QStash and executed by the internal dispatch route `POST /api/internal/jobs/dispatch`.

## Auth setup

Set these credentials in `.env.local` or Vercel project settings:

- `AUTH_SECRET`
- `AUTH_USER_EMAIL`
- `AUTH_USER_PASSWORD`

Optional Google OAuth login:

- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `GOOGLE_REDIRECT_URI` (default local callback: `http://localhost:3000/api/auth/google/callback`)
- `GOOGLE_AUTH_ALLOWED_EMAILS` (comma-separated allowlist, defaults to `AUTH_USER_EMAIL` when unset)

Google Cloud OAuth callback URI must match:

- `https://<your-domain>/api/auth/google/callback` (production)
- `http://localhost:3000/api/auth/google/callback` (local)

Protected surfaces:

- Dashboard route: `/dashboard`
- Job APIs: `GET /api/jobs`, `POST /api/jobs`, `GET /api/jobs/:id`

## Storage Architecture

The system supports three storage backends with automatic detection and fallback:

### Priority Order (Production → Development)

1. **Vercel Blob** (production): Set `BLOB_READ_WRITE_TOKEN`
   - File content stored in scalable Blob storage
   - Metadata stored in Postgres
   - Public URLs for worker access
   - Ideal for Vercel deployments with high concurrency

2. **PostgreSQL** (optional): Set `POSTGRES_URL`
   - Both file content and metadata in Postgres
   - Good for self-hosted or traditional deployments
   - Requires backup strategy for large files

3. **Local JSON** (default): Dev fallback
   - Metadata and content in `.local-data/uploads.json`
   - No external dependencies
   - Perfect for local development

### Configuring Storage

**Development (local)**:
```bash
# .env.local has no BLOB or POSTGRES configured
# System uses .local-data/uploads.json
npm run dev
```

**Staging (Postgres)**:
```bash
POSTGRES_URL=postgresql://...
# System uses Postgres for both jobs and uploads
```

**Production (Vercel + Blob)**:
```bash
# In Vercel project settings:
BLOB_READ_WRITE_TOKEN=vercel_blob_rw_...
POSTGRES_URL=postgresql://...  # For job metadata

# CSV uploads stored in Vercel Blob
# Worker fetches from public blob URLs
```

## Deployment to Vercel

1. **Connect GitHub repo** to Vercel
2. **Set Root Directory** to `frontend` in Project Settings -> Build and Deployment
   - This repository is a monorepo; the Next.js app lives in `frontend/`
   - Avoid legacy `builds`/`routes` overrides for Next.js routing
3. **Set environment variables**:
   - `AUTH_SECRET` (16+ chars)
   - `AUTH_USER_EMAIL`
   - `AUTH_USER_PASSWORD`
   - `GOOGLE_CLIENT_ID` (optional, enables Google button)
   - `GOOGLE_CLIENT_SECRET` (optional, enables Google button)
   - `GOOGLE_REDIRECT_URI` (optional override)
   - `GOOGLE_AUTH_ALLOWED_EMAILS` (optional allowlist)
   - `WORKER_WEBHOOK_SECRET`
   - `WORKER_BASE_URL` (your deployed worker URL)
   - `BLOB_READ_WRITE_TOKEN` (auto-generated by Vercel)
   - `POSTGRES_URL` (optional, for job persistence)

4. **Deploy**: Push to main branch or use Vercel CLI
5. **Test**: Access deployed frontend, login, upload CSV, submit job

## Quality gates

A CI workflow is defined at [ci_cd/.github/workflows/quality-gates.yml](../ci_cd/.github/workflows/quality-gates.yml) and runs:

- **Frontend**: ESLint validation, Jest API integration tests (23 tests), Playwright E2E tests (23 tests), production build
- **Worker**: Python syntax compile check, pytest pipeline tests (3 tests)

### Running tests locally

```bash
# Frontend unit/API tests
cd frontend
npm test                    # Jest API integration tests (excludes E2E)
npm test -- --watch        # Watch mode
npm run lint              # ESLint validation
npm run build             # Production build

# Frontend E2E tests (includes lifecycle tests)
npm run e2e               # Playwright E2E tests (auto-starts dev server)
npm run e2e:ui            # Playwright UI mode for debugging

# Worker
cd ..
python -m pytest tests/unit/test_worker_pipeline.py -v
```
